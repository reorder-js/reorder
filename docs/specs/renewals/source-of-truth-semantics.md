# Reorder: Renewals Source-of-Truth and Renewal Semantics Spec

This document covers step `2.3.3` from `documentation/implementation_plan.md`.

Goal:
- define the source-of-truth rules for renewal execution
- define how subscription snapshots are used during renewal
- define when `pending_update_data` is only preview state and when it becomes executable input
- define when approval is required
- define how `Plans & Offers` affects renewal execution
- define the boundary between subscription state and renewal-cycle state

This specification builds on:
- `reorder/docs/specs/subscriptions/domain-model.md`
- `reorder/docs/specs/plan-offers/effective-config-semantics.md`
- `reorder/docs/specs/renewals/domain-model.md`

The direction follows Medusa patterns:
- one module should have a clear source of truth for its operational state
- workflows should consume explicit domain data, not implicit UI assumptions
- snapshots should be used where historical execution must remain stable
- computed cross-module state should be re-resolved when business validation requires current policy

Implementation status:
- the `Renewals` area is now implemented
- treat this document as design-time context and source-of-truth rationale
- the current runtime source of truth lives in:
  - `reorder/docs/architecture/renewals.md`
  - `reorder/docs/api/admin-renewals.md`
  - `reorder/docs/admin/renewals.md`
  - `reorder/docs/testing/renewals.md`

## 1. Core semantics

The `Renewals` area has three separate concerns:

- subscription operational state
- renewal-cycle execution state
- current commercial policy validation

These concerns must not be mixed.

### Subscription operational state

Subscription operational state lives in the `subscription` module.

It includes:
- current subscription status
- current subscribed product and variant
- current cadence fields
- current shipping address
- current customer, product, and pricing snapshots
- current `pending_update_data`

### Renewal-cycle execution state

Renewal-cycle execution state lives in the `renewal` module.

It includes:
- when a cycle is scheduled
- whether the cycle is processing, succeeded, or failed
- whether approval is required or already decided
- which order was generated by that cycle
- which pending changes were actually applied in that cycle
- what attempts were made to execute the cycle

### Current commercial policy validation

Current commercial policy validation is derived from `Plans & Offers`.

It is not persisted as the long-term source of truth of a subscription or a renewal cycle.

It is re-resolved when business rules require checking whether a renewal is still allowed under the active offer configuration.

## 2. Source of truth by concern

### Subscription source of truth

The source of truth for the customer’s subscription remains `Subscription`.

The `subscription` record owns:
- active cadence
- active product and variant
- lifecycle status
- next renewal scheduling anchor
- current operational snapshots
- pending but not yet applied changes

### Renewal source of truth

The source of truth for renewal execution remains `RenewalCycle`.

The `renewal_cycle` record owns:
- the execution state of one renewal unit
- approval state for that unit
- the resulting renewal order reference
- the applied pending-change snapshot, if any

### Attempt-history source of truth

The source of truth for retry and execution history remains `RenewalAttempt`.

Attempts do not replace the cycle aggregate state.

### Offer-policy source of truth

The source of truth for allowed frequencies and offer rules remains `PlanOffer`.

`ProductSubscriptionConfig` is derived state resolved from `PlanOffer` at read or validation time.

## 3. What data builds a renewal order

The renewal order should be built from the subscription’s current operational state, optionally modified by approved pending changes.

The source set should be:
- current `Subscription` scalar fields
- subscription snapshots needed for stable renewal execution
- subscription shipping address snapshot
- approved and applicable `pending_update_data`
- current `Plans & Offers` effective config only for validation, not as the main order payload source

### Primary source fields

The renewal flow should treat these `Subscription` fields as primary input:
- `subscription.id`
- `status`
- `product_id`
- `variant_id`
- `frequency_interval`
- `frequency_value`
- `shipping_address`
- `customer_snapshot`
- `product_snapshot`
- `pricing_snapshot`
- `pending_update_data`

### Why live reads are not the primary source

Renewal execution should not depend primarily on live reads of:
- customer profile data
- product title data
- variant title or SKU data

Why:
- the subscription already stores operational snapshots
- renewal execution needs stable and repeatable context
- Admin and audit flows should be able to explain what was used during execution

Live reads may still be used:
- for policy validation
- for linked display enrichment
- for internal order-creation dependencies where required by later workflow design

But they should not replace the subscription’s own operational state as the main input.

## 4. How subscription snapshots are used

### Customer snapshot

`customer_snapshot` is used as the stable renewal customer context.

It should support:
- auditability
- Admin readability
- later order-context reconstruction

It is not a substitute for payment-provider customer references if those are needed later.

### Product snapshot

`product_snapshot` is used as the stable subscription commerce snapshot.

It should preserve:
- product title
- variant title
- SKU

This keeps renewal detail and historical cycle detail understandable even if product display data changes later.

### Pricing snapshot

`pricing_snapshot` is the stable subscription pricing context.

It should represent the currently active pricing view on the subscription before a new renewal cycle is executed.

Important note:
- `pricing_snapshot` is not the source of truth for whether a future change is still allowed
- that policy validation belongs to `Plans & Offers`

### Shipping address snapshot

`shipping_address` is the operational fulfillment address for renewal execution.

Renewals should use the subscription’s shipping address snapshot, not the customer’s current global addresses.

## 5. `pending_update_data` semantics

`pending_update_data` belongs to `Subscription`.

It is transitional subscription state, not renewal-cycle state.

### Before renewal execution

Before a cycle executes:
- `pending_update_data` is only a preview of a future change
- it does not yet replace the active subscription cadence or variant
- it may require approval before it becomes executable input

### During renewal execution

During a renewal cycle:
- the workflow determines whether `pending_update_data` is applicable for this cycle
- if applicable and allowed, it becomes candidate execution input
- if applied, the cycle stores a snapshot in `RenewalCycle.applied_pending_update_data`

### After renewal execution

If a renewal succeeds with applied pending changes:
- those changes should be materialized back into the `Subscription` active state
- the subscription’s active fields should be updated
- `pending_update_data` should be cleared or otherwise transitioned out of preview state

If a renewal does not apply pending changes:
- `pending_update_data` remains preview state on the subscription
- the cycle should not pretend those changes were used

## 6. When `pending_update_data` is applicable

`pending_update_data` should only be considered for a cycle when all of the following are true:
- the subscription is otherwise eligible for renewal
- `pending_update_data` exists
- the change is effective for this renewal cycle
- approval requirements are satisfied
- the change still passes offer-policy validation at execution time

This means `pending_update_data` is not automatically executable just because it exists.

## 7. Approval semantics

Approval belongs to the renewal cycle, not to the subscription.

Why:
- approval is a decision about whether this cycle may apply a pending change
- a cycle may require approval even though the subscription itself remains active
- approval is part of execution governance, not part of the long-term subscription state

### Recommended rule

Approval should be required only when a cycle would apply `pending_update_data` and the business policy says that the change must be reviewed before execution.

At this stage, the source-of-truth contract should support:
- `approval_status = null` when no approval is required
- `approval_status = pending` when approval is required and undecided
- `approval_status = approved` when the cycle may apply the change
- `approval_status = rejected` when the cycle must not apply the change

### When approval is evaluated

Approval requirement should be decided at cycle creation time or cycle preparation time, not ad hoc in the UI.

Why:
- the cycle needs explicit operational state
- queue processing should know whether the cycle is blocked
- Admin should read a stable approval status from the cycle

## 8. How `Plans & Offers` affects renewal

`Plans & Offers` should influence renewals as current policy validation, not as the main renewal state store.

### What it validates

Before a cycle applies a pending change, the renewal workflow should re-resolve the effective config and validate:
- that an active offer still exists for the relevant product / variant context
- that the requested frequency is still allowed
- that the change remains valid under the current commercial policy

### What it does not replace

`Plans & Offers` should not replace:
- the subscription as the source of current subscribed state
- the renewal cycle as the source of execution state
- the applied pending-change snapshot recorded on the cycle

### Why current policy should be re-validated

The same change may have been valid when scheduled but invalid by the time renewal executes.

Re-validating against the current effective config keeps renewal execution aligned with the active business policy.

This follows the same logic already used in the subscription plan-change flow.

## 9. No live-policy snapshot as cycle source of truth

The renewal cycle should not persist a full copy of `ProductSubscriptionConfig` as its primary source of truth.

Why:
- `ProductSubscriptionConfig` is derived state
- duplicating the full policy would create synchronization ambiguity
- the cycle only needs to persist what was actually applied, not the entire effective config object

Persist only:
- the applied pending-change snapshot if used
- convenience fields such as approval status and generated order reference

## 10. Boundary between subscription state and renewal-cycle state

### Subscription state

Subscription state answers:
- what the active subscription currently is
- what variant and cadence are currently active
- what future change is scheduled as preview state
- when the next renewal should occur

### Renewal-cycle state

Renewal-cycle state answers:
- what renewal unit is being processed
- whether the cycle is blocked, processing, failed, or succeeded
- whether approval is pending, approved, or rejected
- what happened during execution
- what order, if any, was generated

These are different domains and must not be collapsed into one model.

## 11. What returns to subscription state after success

If a renewal succeeds:
- `last_renewal_at` on the subscription should advance
- `next_renewal_at` should be recalculated

If the cycle applied approved pending changes:
- the subscription’s active `variant_id` should update
- the subscription’s active `frequency_interval` should update
- the subscription’s active `frequency_value` should update
- related snapshots on the subscription may need to be refreshed
- `pending_update_data` should be cleared

The cycle remains the record of execution history, but the subscription remains the long-term source of active state.

## 12. What happens when approval is rejected

If approval is rejected:
- the cycle should not apply `pending_update_data`
- the cycle may still be eligible to renew using the current active subscription state, depending on later workflow policy
- rejection should not silently mutate the subscription’s current active fields

This keeps approval governance explicit and avoids accidental state changes.

## 13. Operational summary

The recommended operational interpretation is:

- `Subscription` = current subscription truth
- `pending_update_data` = future change preview on the subscription
- `RenewalCycle` = one renewal execution record and approval gate
- `RenewalAttempt` = technical execution history
- `PlanOffer` / `ProductSubscriptionConfig` = current policy used for validation

Each concern has its own owner and should remain separate.

## 14. Impact on later steps

This source-of-truth contract implies:
- the final renewal data model should not duplicate the full subscription model
- the renewal workflow must read from the subscription first, then apply approval and policy validation
- the cycle should persist only the change snapshot that was actually applied
- the Admin read model should expose both subscription context and cycle state without confusing them

It also means later implementation should avoid:
- treating `pending_update_data` as already active subscription state
- treating effective config as persisted renewal state
- rebuilding historical renewal context only from live linked entities
