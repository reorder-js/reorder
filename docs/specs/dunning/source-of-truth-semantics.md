# Reorder: Dunning Source-of-Truth and Responsibility Boundaries Spec

This document covers step `2.4.2` from `documentation/implementation_plan.md`.

Goal:
- define the source-of-truth split between `Subscriptions`, `Renewals`, and `Dunning`
- decide what entity anchors a `DunningCase`
- decide whether one subscription may have more than one active dunning case
- decide whether later failed retries update the same case or create a new one
- define which state belongs to the subscription, renewal cycle, and dunning case

This specification builds on:
- `reorder/docs/specs/subscriptions/domain-model.md`
- `reorder/docs/specs/renewals/domain-model.md`
- `reorder/docs/specs/renewals/source-of-truth-semantics.md`
- `reorder/docs/specs/dunning/trigger-entry.md`

The direction follows Medusa patterns:
- each module owns one clear operational aggregate
- cross-domain coordination happens through workflows and links, not overlapping ownership
- one domain record should not duplicate another domainŌĆÖs primary state machine
- read models may enrich across modules, but the aggregate root must remain explicit

Implementation status:
- `Dunning` is implemented
- this document remains a design-time and decision-history specification for aggregate boundaries and ownership
- runtime source-of-truth lives in `docs/architecture/dunning.md`, `docs/api/admin-dunning.md`, `docs/admin/dunning.md`, and `docs/testing/dunning.md`

## 1. Core semantics

The recurring commerce flow has three separate operational concerns:

- ongoing customer subscription state
- execution state of one due renewal cycle
- recovery state of one failed collectible payment event

These concerns must not be merged.

### Subscription operational state

`Subscriptions` own the customerŌĆÖs ongoing recurring contract.

This includes:
- whether the subscription is active, paused, cancelled, or past due
- the active product and variant
- the active cadence
- operational snapshots such as shipping and product context
- pending but not yet applied subscription changes

### Renewal execution state

`Renewals` own one concrete billing execution unit.

This includes:
- when one renewal cycle is due
- whether that cycle is scheduled, processing, succeeded, or failed
- whether approval is required and what was decided
- which order was generated by that cycle
- what attempts were made to execute that cycle

### Dunning recovery state

`Dunning` owns payment recovery after a renewal failed with a payment-qualified failure.

This includes:
- whether the failed debt event is open for recovery
- how many recovery attempts were made
- when the next retry is due
- what the latest payment recovery error is
- whether the debt was recovered or closed unrecovered

## 2. Aggregate-root decision

The aggregate root for `Dunning` should be `DunningCase`.

`DunningCase` should be anchored to one concrete failed debt event originating from one `RenewalCycle`.

This means:
- the primary identity of a case comes from the failed renewal event
- `RenewalCycle` is the originating source event
- order and payment artifacts are supporting references, not the aggregateŌĆÖs primary anchor

## 3. What anchors a `DunningCase`

For MVP, a `DunningCase` should be associated with:

- one `subscription_id`
- one originating `renewal_cycle_id`
- optionally one `renewal_order_id`
- optionally one payment artifact reference such as `payment_collection_id`

### Final decision

The primary anchor is:
- one concrete `renewal_cycle_id`

The secondary references are:
- `subscription_id`
- `renewal_order_id`
- payment artifact references if later needed

Why this is the preferred boundary:
- the debt event is created by a failed renewal cycle, not by a generic subscription condition
- one renewal cycle already represents one due billing unit
- payment collection and payment session are implementation artifacts that may change across retries
- keeping the case anchored to the originating renewal cycle preserves a stable business identity

## 4. Why `DunningCase` should not be anchored primarily to order or payment collection

### Not order-first

`DunningCase` should not be anchored primarily to `renewal_order_id`.

Why:
- some qualifying payment failures may happen before a fully usable order-payment recovery path is stabilized
- one recovery flow may later recreate or adjust payment artifacts
- the business question is ŌĆ£recover the missed renewal debt for this cycleŌĆØ, not ŌĆ£manage this one order as the sole source of truthŌĆØ

Order data is still important, but it is supporting execution context.

### Not payment-collection-first

`DunningCase` should not be anchored primarily to `payment_collection_id`.

Why:
- payment collection is a payment-implementation artifact, not the customer-facing debt event
- later retry strategy may reuse or recreate payment collections or sessions
- coupling the case identity to a mutable payment artifact would weaken the aggregate boundary

Payment collection references may exist on the case for retry orchestration or diagnostics, but they should not define case identity.

## 5. Subscription-level uniqueness

One subscription may have at most one active dunning case at a time in MVP.

### Final decision

Recommended invariant:
- one subscription can have many historical dunning cases over time
- one subscription can have only one active dunning case at any given moment

Active means statuses such as:
- open
- retry_scheduled
- retrying
- awaiting_manual_resolution

Closed means statuses such as:
- recovered
- unrecovered

Why this is preferred:
- a subscription should not have multiple concurrent payment-recovery operators for overlapping debt events in MVP
- Admin handling stays operationally simple
- the renewal scheduler and dunning scheduler avoid competing recovery state for the same subscription

### How this interacts with cycle anchoring

The case is still anchored to one `renewal_cycle_id`.

The uniqueness rule means:
- if a new payment-qualified renewal failure happens while another case for the same subscription is still active, that is an invalid or blocked situation in MVP
- later workflow rules should prevent creating a second active case until the current one is resolved

This is intentionally conservative and keeps the product behavior understandable.

## 6. Retry ownership

Later failed retries should update the same `DunningCase`, not create a new case.

### Final decision

Recommended model:
- `DunningCase` is the aggregate root for one recovery journey
- `DunningAttempt` is the append-only attempt history for that case
- every later recovery attempt belongs to the same case until it closes as `recovered` or `unrecovered`

Why:
- repeated recovery attempts are one operational thread
- Admin detail should show one case timeline
- a new case for every retry would fragment history and complicate metrics, scheduling, and operator review

## 7. Historical multiplicity

One subscription may have many dunning cases over its lifetime.

This happens when:
- one case is closed
- a later renewal cycle produces a new payment-qualified failure
- a new debt event begins after the previous recovery journey ended

This means:
- uniqueness is only for active cases
- historical records remain many-to-one from case to subscription

## 8. State ownership by domain

The domains should own state in the following way.

### 8.1 State owned by `Subscription`

`Subscription` owns:
- lifecycle status such as `active`, `paused`, `cancelled`, `past_due`
- current active variant and cadence
- current next renewal schedule
- shipping and customer/product snapshots
- pending subscription changes

`Subscription` does not own:
- one cycleŌĆÖs execution result
- one cycleŌĆÖs approval result
- payment recovery attempt history
- the retry schedule of dunning

### 8.2 State owned by `RenewalCycle`

`RenewalCycle` owns:
- due date and execution schedule for one billing event
- execution status such as `scheduled`, `processing`, `succeeded`, `failed`
- approval requirement and approval decision for that cycle
- the generated order reference for that cycle
- the execution snapshot of applied pending changes
- latest execution error summary and attempt count

`RenewalCycle` does not own:
- the long-running payment recovery lifecycle after the payment-qualified failure is classified
- the retry schedule of debt recovery
- recovery closure outcome such as recovered/unrecovered

### 8.3 State owned by `DunningCase`

`DunningCase` should own:
- whether the failed debt event is currently open for recovery
- the dunning status machine
- `attempt_count` for dunning retries
- `max_attempts`
- `next_retry_at`
- latest payment recovery error summary
- recovery timestamps such as `recovered_at` or `closed_at`
- operator recovery decision summary where applicable

`DunningCase` should not own:
- the subscriptionŌĆÖs active product or cadence
- the renewal cycleŌĆÖs approval state
- the renewal cycleŌĆÖs scheduled billing date as the main scheduling source
- the source commercial policy from `Plans & Offers`

## 9. Status boundary rules

Statuses from different domains must not be overloaded.

### Subscription status

Subscription status answers:
- what is the current customer-facing operational state of the recurring relationship

Examples:
- `active`
- `paused`
- `cancelled`
- `past_due`

### Renewal cycle status

Renewal cycle status answers:
- what happened to this one due billing execution unit

Examples:
- `scheduled`
- `processing`
- `succeeded`
- `failed`

### Dunning case status

Dunning case status answers:
- what is the current recovery state of the failed collectible payment event

Examples for later steps:
- `open`
- `retry_scheduled`
- `retrying`
- `recovered`
- `unrecovered`

Important rule:
- `DunningCase` status must not replace `RenewalCycle.status`
- `RenewalCycle.status` remains `failed` for the originating failed cycle even if later dunning recovery succeeds

Why:
- the renewal execution history must remain historically true
- dunning recovery is a later operational layer, not a rewrite of the original execution attempt

## 10. Recovery success semantics

If `Dunning` later recovers the payment successfully:

- the originating `RenewalCycle` does not stop being the source of the original failed execution attempt
- the `DunningCase` becomes the source of truth for the recovery outcome
- the `Subscription` becomes the source of truth for any resulting operational state update after recovery

This means a later successful recovery may update:
- subscription status or flags if needed by the business rules
- dunning case status to `recovered`

But it should not mutate history into:
- `RenewalCycle.status = succeeded` retroactively for the original failed attempt

## 11. Read-model boundaries

Admin read models should respect the aggregate roots.

### Dunning list/detail root

The Admin read root for future `Dunning` pages should be `DunningCase`.

It may enrich with:
- subscription summary
- originating renewal cycle summary
- linked order summary
- selected payment context

But those linked records are enrichment only.

### Why this matters

This mirrors the pattern already used in:
- `Subscriptions`, where the root is `Subscription`
- `Renewals`, where the root is `RenewalCycle`

So for `Dunning`:
- the root should be `DunningCase`
- `DunningAttempt` should be child history

## 12. Workflow boundary

Cross-domain workflows should coordinate domains without blurring ownership.

Recommended direction:
- a renewal workflow detects a payment-qualified failure
- a dedicated `start-dunning` workflow creates or updates the dunning aggregate
- dunning retry workflows mutate only dunning-owned state plus explicit downstream payment operations
- if a recovery outcome requires subscription updates, that should happen explicitly in workflow steps

This keeps:
- `Renewals` responsible for renewal execution
- `Dunning` responsible for recovery execution
- `Subscriptions` responsible for subscription lifecycle state

## 13. Final decision summary

For step `2.4.2`, the final decisions are:

- `DunningCase` is a separate aggregate root
- `DunningCase` is anchored primarily to one originating `renewal_cycle_id`
- `renewal_order_id` and payment artifacts are supporting references, not the primary identity of the case
- one subscription may have many historical dunning cases, but only one active case at a time in MVP
- later failed recovery retries update the same `DunningCase` and create `DunningAttempt` history, not new cases
- `Subscription` owns recurring relationship state
- `RenewalCycle` owns one billing execution unit
- `DunningCase` owns payment recovery state for one failed collectible debt event
